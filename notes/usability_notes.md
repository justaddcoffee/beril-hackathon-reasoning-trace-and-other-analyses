# Usability notes

How smoothly did sessions run, regardless of what they produced? Opt-in
main sessions only (252 sessions, 32 users). Generated by
`scripts/01g_usability.py`.

These notes went through an independent codex re-derivation pass; the
corrections it forced are recorded inline so the reasoning is auditable.

## Headline: errors are common but mostly recoverable

Tool calls fail fairly often, but the agent usually retries and succeeds,
so the raw failure rate overstates the friction.

| tool | calls | failure rate | recovery rate † |
|---|---:|---:|---:|
| Bash | 6199 | 12.5% | 73% |
| Read | 1965 | 8.1% | 80% |
| Write | 657 | 6.1% | 91% |
| Edit | 664 | 4.8% | 83% |
| NotebookEdit | 321 | 2.2% | 100% |

† recovery rate = after a tool errors, the next call to that *same* tool in
the session succeeded. It's a lower bound: a recovery via a different tool
isn't counted.

Overall execution-tool failure rate is ~9%. Bash is the noisiest at 12.5%,
but 73% of Bash errors are followed by a successful Bash call. Only 9 Bash
errors in the whole opt-in set were never retried.

See `notes/figures/12_tool_failure_vs_recovery.png`.

## Session-level friction prevalence

| signal | share of opt-in sessions |
|---|---:|
| any tool error | 51% |
| context reset | 14% |
| ended on an unresolved error | 6% (16/252) |
| Bash thrashing (same cmd ≥3×) | 2% |

- **Ended on error**: 16 sessions. As a share of sessions that used any
  tool (157) that's 10%; as a share of all 252 it's 6%. The figure uses the
  /252 denominator.
- **Context resets**: 92 "continued from a previous conversation that ran
  out of context" sentinels across 36 sessions. The context window was a
  more visible friction than tool failures, since a reset is disruptive
  whereas a recovered tool error usually isn't.
- **Thrashing**: only 5 sessions repeated the same Bash command ≥3× in a
  row, and those were mostly notebook-execution retries and git operations,
  not pathological loops.

See `notes/figures/13_usability_friction_overview.png`.

## What the Bash failures actually are

Of 777 Bash errors (categorization is regex-based and noisy — treat as
rough): ~24% are the agent's own Python crashing (`NameError`,
`ModuleNotFoundError`, etc.), ~14% missing files, ~10% user
rejection/interruption strings, ~2% missing commands, the rest bare
`Exit code N` bodies. Recurring infra strings: `No module named
'spark_connect_remote'` (24 errors) and MinIO credential failures (2). So
friction is part agent-mistake, part platform rough-edge, but the clean
split implied by the first cut of this analysis didn't hold up under
scrutiny.

## Corrections forced by the codex review

Recording these so the analysis stays honest:

1. **The "bootstrap retry tax" was dropped.** An earlier cut claimed that
   because 20/32 users ran `/berdl_start` ≥2×, the environment bootstrap
   was a major friction point. That doesn't hold: operationalized properly
   (a bootstrap session that did no work, immediately followed by another
   bootstrap within 10 min), there were only **2 such events across 1
   user**. The high bootstrap counts mostly reflect normal new-session
   starts and heavy-user volume, not a broken environment. Not a finding.

2. **Bash failure composition numbers were revised down.** Python
   exceptions are ~24% (an earlier cut said ~39%), and ~10% of "other"
   were actually user-rejection strings that shouldn't have been lumped in.

3. **AskUserQuestion / ExitPlanMode `is_error` are not tool failures.** They
   are dominated by user rejections. (One AskUserQuestion rejection even
   carried the user's redirect instruction inline.) AskUserQuestion does
   have ~29 genuine `InputValidationError` cases — the agent offering >4
   options — which are real but separate from execution-tool failure. These
   interaction tools are excluded from the failure/recovery figure.

4. **Time-to-first-project-write regex was leaky** — it matched
   `.claude/projects/` memory paths. Tightened, the median is ~5.9 min over
   ~92 sessions (vs a looser 5.5 min / 93 sessions). Directionally stable.

5. **Disengagement: 0/32 opt-in users** had zero engaged sessions (an
   engaged session = ≥3 turns and ≥1 tool call). This corrects an earlier
   claim that "some users opened a session and never engaged" — that
   conflated empty *individual sessions* with whole *users*.
